## Безопасность

- [x] 🔴 **S1 — PII пользователей пишется на диск и 141 файл уже закоммичен в git** *(исправлено 2026-07-13)*
  `LlmRawStore.kt`, включено в `ServerModule.kt:172,176`. Каждый промпт/ответ (возраст/пол/рост/вес/травмы/health notes) пишется в `server/logs/llm/*` без ограничения хранения и без редакции. `.gitignore` эту папку не покрывает.
  Сделано: `git rm -r --cached server/server/logs` (141 файл, коммит `bc8d258`) + добавлено в `.gitignore` + `LlmRawStore` выключен по умолчанию (`LLM_RAW_STORE_ENABLED` env-флаг, off если не задан).
  **Не сделано:** история git всё ещё содержит эти файлы в старых коммитах — нужен `git filter-repo` + force-push, это разрушительная операция по всей истории репозитория, требует явного отдельного решения пользователя, не выполнялась автономно.

- [ ] 🔴 **S2 — подписка не работает и не проверяется сервером** *(оценено 2026-07-13, не исправлено — нужны внешние действия)*
  `AndroidPurchasesRepository.kt:25-32` всегда `false` (Android). `StubPurchasesRepository` (`isSubscriptionActive() = true`) — дефолт в общем `KoinModule.kt`, используется на iOS (нет override) → на iOS всегда "премиум" бесплатно. `/ai/*` эндпоинты (`AiRoutes.kt`) вообще не проверяют подписку — подтверждено, нуль упоминаний subscription/entitlement в серверном коде.
  Известный вектор атаки на общий секрет — подробности намеренно не публикуются в открытом репо, см. внутренние заметки/обсудить с командой напрямую.
  **Почему не исправлено:** реальный фикс требует (1) настроенных продуктов/SKU в Google Play Console, (2) service account + доступ к Google Play Developer API для серверной верификации токена покупки, (3) per-user идентификации (упирается в S3 — сейчас только общий секрет, не настоящая авторизация). Всё это — внешние настройки аккаунта, которые не может сделать агент. Частичный фикс (например, замена клиентского stub на billing-запрос к несуществующему SKU) хуже текущего состояния: либо тихо ломает paywall всем, либо создаёт ложное чувство защищённости — клиентская проверка в любом случае не защищает от прямого вызова API с токеном из APK.
  Нужно от владельца: создать продукт подписки в Play Console, получить service account credentials для Developer API, затем реализовать server-side проверку + связать с S3 (per-user auth).

- [x] 🔴 **S3 — единый статичный секрет в APK, сравнение не timing-safe** *(исправлено 2026-07-13)*
  `Application.kt:51-63`, токен зашит в `app-android/build.gradle.kts:53`. R8 не защищает строковую константу — достаётся `strings`/`apktool`. Сравнение `token != appSecret` — обычное, не constant-time.
  Сделано: `MessageDigest.isEqual` вместо `!=` (коммит `bc8d258`).
  **Не сделано:** переход на per-user auth (Firebase Auth ID tokens) — общий секрет остаётся, это больше архитектурное решение, завязанное на S2 выше.

- [x] 🟡 **S4 — CORS открыт на любой origin** *(исправлено 2026-07-13)*
  `Application.kt:36` — `install(CORS) { anyHost() }`. Мобильному бэкенду не нужен permissive CORS вообще.
  Сделано: `install(CORS)` убран полностью (нативные клиенты Origin не шлют).

- [x] 🟡 **S5 — рейт-лимит только по IP, обходится/коллапсирует за прокси** *(исправлено 2026-07-13)*
  `Application.kt:42-46` — ключ `origin.remoteHost`, нет `XForwardedHeaders`. За Cloud Run может схлопнуться на всех пользователей или обходиться подделкой `X-Forwarded-For`.
  Сделано: `install(XForwardedHeaders) { useLastProxy() }` — важно именно `useLastProxy()`, без него плагин доверяет ПЕРВОМУ (клиентскому, подделываемому) значению заголовка, что делает рейт-лимит ещё более обходимым, чем до фикса (поймано на ревью Fable 5).
  **Не сделано:** реальная per-user квота вместо per-IP — упирается в то же отсутствие per-user identity (S2/S3).

- [x] 🟠 **S6 — prompt injection через свободный текст профиля (травмы/health notes/чат)** *(исправлено 2026-07-13)*
  Летел в промпт без экранирования: `ChatService.kt`, `AiService.kt`, `CoachBundlePromptBuilder.kt`, `InjuryRestrictions.kt`, `AiNutritionSafetyPolicy.kt`. Самое опасное место — сырая строка травмы внутри "NON-NEGOTIABLE" блока в `InjuryRestrictions.kt` и сырые customTerms внутри "ABSOLUTE DIETARY RESTRICTIONS" в `AiNutritionSafetyPolicy.kt`.
  Сделано: новый `PromptSafety.kt` — `untrustedDataBlock()` (полный блок с `<<<USER_DATA_START/END>>>` для крупного контента типа PROFILE JSON) и `sanitizeInlineUserText()` (для коротких полей: чистит все варианты переноса строки включая Unicode-разделители, вырезает `<`/`>` целиком чтобы нельзя было пересобрать маркер вложенностью). Применено на всех найденных сайтах инъекции сырого текста профиля/чата.
  Severity была средняя, не критичная — серверный нормализатор/`InjuryFilter`/`validateBundle` всё равно авторитетно пересчитывают структуру плана независимо от того, что скажет модель; фикс закрывает поверхность на свободном текстовом выводе (реплики чата, советы, названия блюд), который НЕ перевалидируется структурно.

- [x] 🟠 **S7 — `.env` ищется по 8 путям + названия ключей печатаются в stdout** *(исправлено 2026-07-13)*
  `Env.kt:15-24,29`. В проде не используется (Cloud Run secrets), но само поведение — сузить список путей, убрать println имён ключей или сделать debug-level.
  Сделано: список путей сужен до 2 (repo root + `server/` — единственные реально используемые), `println` заменён на SLF4J `logger.debug` с количеством ключей вместо их имён.

## Архитектура и расширяемость

- [ ] 🟠 **A1 — упражнение существует в 3 несовместимых списках без единого источника правды**
  Сервер (188, `BuiltInExerciseCatalog.kt`) / клиент detail (104, `ExerciseLibrary.kt`) / клиент browse (107, `ExerciseBrowseCatalog.kt`). Сервер может отдать ID, для которого на клиенте нет техники/картинки — пользователь видит голый лейбл.
  Фикс: единый источник (кодген из одного файла или сервер читает `shared`-каталог), минимум — тест "все ID сервера есть в `ExerciseLibrary`".

- [ ] 🟠 **A2 — allow/deny списки упражнений (seconds/minutes/bodyweight) — 4-я копия тех же фактов**
  `AiTrainingPlanPolicy.kt:21-71` (`durationSecondsExerciseIds`, `durationMinutesExerciseIds`, `bodweightOnlyExerciseIds`) дублируют то, что логически должно быть полем в `ExerciseCatalogItem`; клиент независимо переизобретает то же самое через `ExerciseCalibrationKind`. Это уже реально дрейфовало (см. git log `2a1e2a3`, `09e7ae7`).
  Фикс: перенести `isBodyweightOnly`/`durationUnit` на сам каталог, удалить отдельные сеты.

- [x] 🟠 **A3 — TDEE/макросы посчитаны по-разному в 2+ местах** *(частично исправлено 2026-07-14)*
  Сервер: богатая модель (`CoachBundlePromptBuilder.computeTargetNutrition`). Клиент офлайн-фолбек: `NutritionRepositoryDb.tdeeKcal()` с фиксированной активностью 1.4, без trend/lifestyle. При отказе сети пользователь получает другую цель по калориям, молча.
  Идеальный фикс (вынести в `shared/domain` как единую чистую функцию) заблокирован реальным архитектурным ограничением: `:server` не имеет Gradle-зависимости на `:shared`, а `:shared` объявляет только `androidTarget()`/`iosX64()`/`iosArm64()`/`iosSimulatorArm64()` — без plain `jvm()`, который мог бы использовать `:server`. Добавление jvm-таргета в `:shared` (тянущего Compose/SqlDelight/Ktor-client/Napier) — это реальное изменение топологии сборки, слишком рискованное для точечного фикса.
  Сделано (прагматично): портировал ТЕ ЖЕ константы и формулу (lifestyle activity factor, training-day bonus, goal calorie factor, macro multiplier ranges) в клиентский `tdeeKcal`/`macrosFor` — теперь оба места считают одинаковые числа для одинаковых входов (код всё ещё физически дублирован, с комментарием-ссылкой на серверную функцию). Заодно исправлен расхождение: клиент использовал fat=0.8×вес вместо серверных 0.9×вес. Trend-adjustment (по `recentWeights`) сознательно НЕ портирован — эти данные недоступны в офлайн-слое клиента, задокументировано в коде. Проверено Fable 5 построчным сравнением констант, тесты зелёные.

- [x] 🟠 **A4 — fallback-генераторы плана — второй, вручную синхронизируемый "движок"** *(исправлено 2026-07-14)*
  `AiFallbackFactory.kt` (уже источник багов — `ae5b6b9`, `85edb5a`). Частично лечится через `normalizeTrainingPlan`/`computeTargetNutrition`, но `fallbackTraining` держал свой хардкод-шаблон из 3 паттернов (цикл `day % 3`, магические веса 60/45/90кг), не учитывающий реальный сплит/опыт/кол-во дней пользователя.
  Сделано: `fallbackTraining` теперь строит `WorkoutSkeleton` через тот же `TrainingSplitPlanner.build(...)` с реальными параметрами профиля (фокус/цель/сплит/опыт/травмы/deload), что и основной путь через `computeSkeletonData` в `normalizeTrainingPlan` — слоты/паттерны для нестандартных сплитов (5-6 дней PPL и т.д.) больше не рассинхронизированы с реальным количеством. `weightKg` в фолбеке честно `null` (без LLM нет способа оценить вес пользователя) вместо вымышленных чисел — UI показывает "—". Проверено Fable 5, тесты зелёные.

- [x] 🟠 **A5 — `AiService.kt` (832 строки) — God object** *(исправлено 2026-07-13)*
  Кэш + оркестрация + промпт-билдеры + валидация в одном классе; free/paid fallback-политика (`shouldFallbackToFree` и т.д.) буквально скопирована в `ChatService.kt:219-247`.
  Сделано: `BundleCache` вынесен (см. N1), промпт-билдеры → `SectionPromptBuilders.kt`, валидация бандла → `BundleValidation.kt`, LLM-роутинг (`generateWithFallback`, тоже дублировался в `ChatService.kt`) → общий `LlmClientRouter.kt`. `AiService.kt`: 832 → 511 строк. Три отдельных коммита, каждый проверен компиляцией/тестами и ревью Fable 5.

- [x] 🔵 **A6 — мёртвый код в DI + брошенные `SignupRequest`/`SignupResponse`** *(исправлено 2026-07-13)*
  `KoinModule.kt` (незарегистрированные репозитории, уже в RELEASE_CHECKLIST.md), `Api.kt:161-164` — DTO для auth-флоу, которого нигде больше нет (намёк на брошенную идею per-user auth — корень S2/S3).
  Сделано: удалены `SignupRequest`/`SignupResponse` (нулевые ссылки нигде в коде) и `ProfileSettingsRepository.kt` (абстрактный класс, ~85 строк, тоже нулевые ссылки — RELEASE_CHECKLIST.md называл другие 3 класса, которых уже нет в коде; это тот же паттерн, найденный заново).
  **Не сделано:** реализовать auth, на который намекали DTO — это S2/S3, требует внешних действий (см. выше).

## Неоптимальные решения

- [x] 🟡 **N1 — кэш в `companion object` ломается при горизонтальном масштабировании** *(частично исправлено 2026-07-13)*
  `AiService.kt:806-820`. `--max-instances=5` в `cloudbuild.yaml` → 5 независимых кэшей и 5 независимых in-flight dedup карт. N одинаковых запросов на разные инстансы = N дорогих вызовов Sonnet вместо одного.
  Сделано: кэш вынесен в отдельный класс `BundleCache.kt`, зарегистрирован как Koin `single`. В рамках ОДНОГО процесса поведение идентично старому (AiService и так уже был Koin `single`) — выигрыш в том, что это теперь явный, тестируемый коллаборатор, а не JVM-статик, который незаметно шарит состояние между любым `AiService`, созданным в обход Koin (например, в тестах).
  **Не сделано:** кросс-инстансный дедуп между 5 репликами Cloud Run — по-прежнему требует Redis/Memorystore, это отдельная инфраструктурная задача, не запускалась (аудит сам называет её опциональной, "если это реально важно по деньгам").

- [x] 🟠 **N2 — непоследовательная стратегия обработки ошибок** *(сервер исправлен 2026-07-13 через FIX_PLAN Wave 1.4, клиент — нет)*
  29 мест `runCatching`/`getOrElse`, много "любой Throwable → тихий фолбек" (`AiService.kt:88-94`, `ChatService.kt:44-89`, `PaywallViewModel.kt:20`). Реальные баги (NPE, рассинхрон сериализации) неотличимы от "LLM подтормозил" — оба тихо превращаются в фолбек-план. Вероятно, именно это давало багам этой сессии время жить незамеченными.
  Сделано: `AiService.kt`/`ChatService.kt` теперь ловят только ожидаемые типы (`LlmException.*`, `LlmPipelineExhaustedException`, `TimeoutCancellationException`) через `isExpectedLlmFailure()`; всё остальное пробрасывается как есть (500), не тонет в фолбеке.
  **Не сделано:** `PaywallViewModel.kt:20` (`runCatching { validateSubscription() }.getOrDefault(false)`) — клиентский код, не трогали. Низкая severity (просто UI-флаг подписки), но тот же паттерн.

- [x] 🟠 **N3 — управление потоком по substring-мэтчингу текста ошибки** *(исправлено 2026-07-13)*
  `AiService.kt:780-804`, дублируется в `ChatService.kt:236-247`, `OpenRouterLLMClient.kt:158-175` — `message.contains(" 429")` и т.п. вместо типизированных ошибок.
  Сделано: новая sealed-иерархия `LlmException` (RateLimited/PaymentRequired/Auth/Timeout/Provider), `OpenRouterLLMClient` мапит HTTP-статус на границе, `shouldTryNextModel`/`RetryingLLMClient.isRetryable`/`shouldFallbackToFree` ветвятся по типу; строковый матчинг остался только как legacy-fallback для нетипизированных случаев.

- [x] 🔵 **N4 — `sanitizePayload` слепо режет между первой `{` и последней `}`** *(исправлено 2026-07-13)*
  `Api.kt` ~`:130-140`, применялся к каждому успешному ответу безусловно — костыль под сервер, у которого уже есть свой надёжный `ResponseExtractor`.
  Сделано: `sanitizePayload` вызывается только как повторная попытка после `SerializationException` при обычном разборе, не перед каждым парсингом.

- [x] 🟠 **N5 — рассинхрон промпта и enforcement по магическим числам** *(исправлено 2026-07-13)*
  Промпт говорил "Max 5 workouts" (`AiService.kt:560`), нормализатор реально разрешал 7 (`MaxWorkoutsPerPlan=7`). Промпт говорил про планку "20–120 сек", а `SECONDS_RANGE=20..60` в `AiTrainingPlanPolicy.kt` молча обрезал 90-секундную планку до 60. Плюс найден отдельный, более серьёзный рассинхрон: промпт описывал bike/cycling как "seconds, 60-600", а код клэмпит их как MINUTES (`durationMinutesExerciseIds`, 5-30) — разные единицы измерения, не просто разные диапазоны.
  Сделано: `MaxWorkoutsPerPlan`, `MaxSetsPerWorkout`, `SECONDS_RANGE`, `MINUTES_RANGE` подняты с `private` до `internal` в `AiTrainingPlanPolicy.kt`, промпт-билдер в `AiService.kt` теперь интерполирует эти константы напрямую вместо хардкода — рассинхрон больше физически невозможен. Также поправлена группировка exercise ID по типу (bike/cycling/rowing_machine → минуты, mountain_climber → секунды, как в коде).

- [x] 🔵 **N6 — `versionCode` захардкожен, `allowBackup=true` при health-данных в БД** *(частично исправлено 2026-07-13)*
  `app-android/build.gradle.kts:50` (уже P2 в RELEASE_CHECKLIST.md — CI-инкремент). Плюс: `allowBackup="true"` в манифесте позволяет вытащить локальную SQLite (профиль, травмы, health notes) через `adb backup` на части устройств.
  Сделано: `android:allowBackup="false"` в манифесте.
  **Не сделано:** `versionCode` из CI (не тронуто — вне контекста бэклога безопасности, отдельная CI/CD задача из RELEASE_CHECKLIST.md).

## Что оставить как есть (проверено, работает хорошо)

- Серверная нормализация skeleton-плана / `InjuryFilter` / `validateTrainingPlan` — LLM трактуется как недоверенный источник, вывод авторитетно пересчитывается. Именно это держит S6 на среднем уровне, а не критичном.
- `NutritionTargetCalculator.kt` — тонкий общий делегат, специально сделан, чтобы прод и тесты шарили одну реализацию.
- `DataResult` sealed-тип + `ApiClient.mapApiThrowable` — образцовая типизация ошибок, к которой стоит привести N2.
- 3-tier LLM pipeline (`LlmPipeline`, extractor/sanitizer/decoder/feedback) — хорошо декомпозирован сам по себе, проблема в оркестрации вокруг (`AiService` God object, N1), не в пайплайне.
- `BootstrapCoachData` mutex — корректный Koin `single` с double-check после лока.
- Условное подключение Firebase-плагина без `google-services.json` — прагматично, задокументировано, не ломает сборку.
