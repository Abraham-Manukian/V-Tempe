# Отчёт тестирования сервера — 8 новых граничных сценариев

**Дата:** 2026-07-09
**Сервер:** локальный (`http://127.0.0.1:8081`), актуальный код со всеми фиксами сессии
(включая сегодняшний фикс `bodweightOnlyExerciseIds` для новых упражнений каталога)
**Метод:** прямые запросы к `/ai/bootstrap` (`X-App-Token`), 8 профилей подряд

---

## ⚠️ Важное предупреждение до результатов

Во всех 8 запросах retry-вызов к OpenRouter внутри пайплайна падал с ошибкой
`{"success": false, "error": "Access denied by security policy."}` — это не наш код
(формат ответа не похож на стандартную ошибку OpenRouter, больше похоже на блокировку
локальным security-софтом/прокси конкретно retry-запроса). Из-за этого **во всех 8
сценариях сервер исчерпал попытки и ушёл в `fallbackTraining()`/`fallbackNutrition()`**
(статичный шаблон) вместо реального ответа LLM. Повторный одиночный запрос (сценарий 1,
после паузы) дал тот же результат — блокировка не случайная/rate-limit, а стабильная для
этой машины/сети.

**Следствие:** ниже — это стресс-тест самого fallback-слоя (сети безопасности "на случай
если LLM недоступен"), а не типичный ответ для живого пользователя. Но именно это и
обнажило нетривиальные баги в самом fallback, которые в норме скрыты, потому что LLM
почти всегда отвечает успешно. Ценность отчёта не пострадала — наоборот, вскрылись баги,
которые не были видны в прошлом тесте (2026-07-02), где LLM отвечал нормально.

## 🔴 Баг #1 (общий, критичный) — `fallbackTraining()` игнорирует расписание пользователя

`server/.../service/AiFallbackFactory.kt:19` — `val workouts = List(3) { ... }` — количество
тренировок **жёстко закодировано как 3**, независимо от `weeklySchedule`.

Подтверждено на реальных данных:
- Сценарий 2 (4 дня: Пн/Вт/Чт/Пт, UPPER_LOWER) → пришло **3** тренировки (Upper A/Lower A/Upper B).
- Сценарий 3 (5 дней: Пн/Вт/Ср/Пт/Сб) → пришло **3** тренировки (Push/Pull/Legs), нет второго
  Push/Legs, которые должны быть при PPL на 5 дней.
- Сценарий 6 (6 дней, явный PPL) → пришло **3** тренировки (Push/Pull/Legs один раз, не два).

Все результаты проходят через `normalizeTrainingPlan()`, который переопределяет
`exerciseId`/`label` из skeleton (`computeSkeletonData`), поэтому подобранные упражнения
выглядят "нормально" (реальные ID из каталога, не заглушки `pattern:knee_dominant`) — баг
не бросается в глаза, но реальное количество тренировочных дней не соответствует тому, что
выбрал пользователь.

**Почему это важно:** это единственная сеть безопасности, когда LLM недоступен (как здесь).
Если такое случится у реального пользователя — вместо честного 6-дневного плана он получит 3
дня и не поймёт, куда делась половина недели.

## 🔴 Баг #2 (общий, критичный) — `fallbackNutrition()` не учитывает целевые калории

`server/.../service/AiFallbackFactory.kt:48` — `templateMeals()` возвращает **фиксированный**
набор блюд с фиксированной калорийностью, не масштабируется под `computeTargetNutrition()`.

Подтверждено: во всех 8 сценариях (от 52кг набора массы до 104кг похудения) итог **абсолютно
одинаковый**: Пн–Сб по 1968 ккал, Вс — 1408 ккал. Реальные расчётные цели по логам сервера
были 2758/2702/2823 ккал (разные для разных профилей) — фолбек попадает мимо цели на
26–34% систематически, и одинаково для противоположных целей (набор массы vs похудение).

Это ровно тот же класс бага, что чинили в этой сессии для LLM-пути (`AiNutritionSafetyPolicy`),
но фолбек-шаблон вообще не проходит через этот механизм — он статичен по определению.

**Рекомендация:** `fallbackNutrition()` должен хотя бы линейно масштабировать порции
шаблона под `computeTargetNutrition(profile)`, как это теперь делает retry-инструкция
для LLM-пути (`buildSevereCalorieFixInstruction`).

## 🔴 Баг #3 (сценарий 8, красный флаг подтверждён) — свободнотекстовые исключения упражнений не работают

Профиль: outdoor, плечо болит, **явно указано** "нельзя подтягивания широким хватом",
"нельзя брусья". Экипировка была указана как `турник, брусья` (это то, что физически
доступно на площадке — сам пользователь пишет это в разделе "Оборудование"), но отдельно в
травмах прямо запрещает брусья.

**Результат:** в плане "Chest + Triceps" присутствует `dip` (упражнение на брусьях,
`server/.../BuiltInExerciseCatalog.kt:184`, требует equipment `brusья`/dip bars) —
ровно тот красный флаг, который просил проверить пользователь.

**Root cause:** `InjuryFilter.kt` понимает только 4 категории тела — колено/плечо/спина/локоть
(`KNEE_KEYWORDS`, `SHOULDER_KEYWORDS`, `BACK_KEYWORDS`, `ELBOW_KEYWORDS`) — и банит по ним
**паттерны движений** (например, "плечо" → банит только `VERTICAL_PUSH`, жим над головой).
Свободнотекстовые исключения конкретных упражнений ("нельзя брусья", "нельзя подтягивания
широким хватом") **не парсятся вообще** — этот текст доходит только до LLM-промпта как
сырая строка и держится исключительно на том, что модель сама прочитает и учтёт его. Когда
генерация уходит в фолбек (как здесь) — эта информация теряется полностью, потому что
фолбек не обращается к LLM вообще.

**Рекомендация:** либо расширить `InjuryFilter` явными keyword→exerciseId исключениями
(проще всего: если строка травмы содержит "брусья"/"dip bars" — исключать `dip`/`bench_dip`
из выбора; если содержит "широким хватом" — исключать `wide_pullup`), либо явно передавать
такие текстовые ограничения в фолбек-путь тоже (сейчас `fallbackTraining` вообще не смотрит
на `req.profile.injuries`, кроме передачи в `normalizeTrainingPlan`, которая гоняет их только
через `InjuryFilter`).

## 🟡 Баг #4 (сценарий 2, средний) — макросы не пересчитываются после замены ингредиентов

`sanitizeTemplateMealsForRestrictions()` корректно убирает курицу/лосось/мёд/орехи из
названия и списка ингредиентов для веган-профиля (проверено — ни одного нарушения текста),
**но kcal/macros блюда остаются от исходного (мясного) рецепта**.

Пример (сценарий 2, обед понедельника):
```
До:  "Куриная грудка с рисом и овощами" — курица, рис, брокколи, морковь — 520 ккал, 45г белка
После: "Сбалансированный обед" — овощи, рис, брокколи, морковь (курица вырезана) — 520 ккал, 45г белка (!)
```
45г белка из "овощи+рис+брокколи+морковь" без курицы — нереалистично завышено. Суммарно за
день получается 145г белка, хотя реальный растительный состав такой белок не даёт.

**Рекомендация:** при удалении ингредиента в `sanitizeTemplateMealsForRestrictions` также
корректировать `macros.proteinGrams`/`fatGrams` (например, вычитать усреднённый вклад
удалённого ингредиента), а не оставлять цифры от оригинального рецепта.

## 🟠 Баг #5 (сценарий 4, мелкий) — пустой `label` у 3-й тренировки

Сценарий 4 (60 лет, много травм, Full Body 2 дня по расписанию — но фолбек всё равно выдал
3 тренировки, см. баг #1): третья тренировка пришла с `"label": ""` (пустая строка) —
на экране пользователь увидит тренировку без названия. Не проверял отдельно, баг ли это
`computeSkeletonData` (когда skeleton короче, чем 3 фолбек-дня) или дефолт в
`normalizeTrainingPlan`.

## Что подтверждено рабочим (несмотря на fallback everywhere)

1. **RPE-клэмп при делоуде (фикс этой сессии) — работает даже через фолбек.** Сценарий 3:
   входные `completionRate 0.58–0.6`, `averageRpe 9.2–9.4` (оба триггера); итоговый план —
   RPE 6.5 на основных движениях, 6.0 на изоляции/кора. Это ровно `RPE_DELOAD_PRIMARY`/
   `RPE_DELOAD_ISOLATION`. Раньше (отчёт от 07-02) точно такой же сценарий давал плоские 8.5
   без единого снижения — фикс подтверждён на новых данных.
2. **Диетические ограничения (веган/аллергии) — текстовая часть чистая.** Ни в одном из 8
   сценариев нет нарушений: сценарий 2 (веган+соя/орехи/арахис/молоко/яйца запрещены) —
   ни одного упоминания курицы/рыбы/сои/тофу/орехов/арахиса/мёда/молока/яиц во всём меню.
3. **Бодивейт-упражнения не получают вес (фикс сегодняшнего дня) — подтверждено на новых
   упражнениях.** Во всех сценариях `pullup`, `chin_up_hold`, `towel_row`, `table_row`,
   `negative_pullup`, `single_leg_glute_bridge`, `knee_pushup`, `bench_dip` и т.п. идут с
   `weightKg: null` — именно те ID, что были добавлены в `bodweightOnlyExerciseIds` сегодня.
4. **InjuryFilter на паттерн-уровне работает.** Сценарий 4 (грыжа+колени+плечо, deadlift
   запрещён по паттерну HINGE, overhead press по VERTICAL_PUSH) — ни deadlift, ни ohp,
   ни jump/burpee в плане нет. Сценарий 7 (колени после смены) — нет прыжков/спринтов.
5. **Список покупок без дублей** — во всех сценариях `shoppingList` дедуплицирован и
   отсортирован по алфавиту, артефактов вида `"Сахар + + — 60 г"` не найдено (это было
   исправлено раньше, регрессии нет).

## Рекомендация по приоритету

1. **Сначала разобраться, почему retry-запрос к OpenRouter получает "Access denied by
   security policy"** на этой машине/сети — до этого нельзя доверять живым (не-фолбек)
   тестам локально. Возможно стоит гонять такие тесты против задеплоенного Cloud Run
   сервера вместо локального (как было в отчёте от 07-02, где такой ошибки не было).
2. **Баг #1 (фолбек-тренировка игнорирует расписание)** и **#2 (фолбек-питание игнорирует
   калории)** — оба легко фиксятся (масштабировать под `weeklySchedule.size` и
   `computeTargetNutrition`), и оба бьют по production reliability при любом сбое LLM,
   а не только по качеству ответа.
3. **Баг #3 (свободнотекстовые исключения упражнений не enforced)** — потенциально самый
   опасный с точки зрения травмобезопасности пользователя, стоит поднять приоритет.
4. Баг #4 (макросы) и #5 (пустой label) — менее критично, косметика/точность данных.
